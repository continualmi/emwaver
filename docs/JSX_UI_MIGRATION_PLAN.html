EMWaver UI Runtime Plan

Move script UI authoring to JSX, keep native rendering.

EMWaver should make scripts feel familiar to write without turning the app into a browser. JSX becomes the authoring layer, with normal-looking HTML-style tags where they map cleanly; the normalized UI tree remains the private bridge from one script runtime to SwiftUI, Android, Windows, Agent tooling, and tests.

Core Decision

Do not migrate to general HTML/CSS or React DOM. Migrate to an EMWaver JSX layer that looks close to normal JSX/HTML for common UI structure, then compiles/evaluates into the current cross-platform UI node model. The user writes familiar code; the native apps render a neutral description of the current UI frame.

Authoring source

JSX-like .emw code

Humans and agents write components, state, event handlers, hardware calls, and conditional UI with a familiar syntax. 

render(() => (
  <main padding={16} gap={12}>
    <h2>GPIO Control</h2>
    <button onClick={() => GPIO.write(1, true)}>
      Set High
    </button>
    <Plot source="samplerBits" height={240} />
  </main>
));
Runtime bridge

Normalized EMWaver UI tree

The script runtime evaluates the code and emits a platform-neutral tree. Renderers consume this private contract; users do not hand-author it. 

{
  type: "column",
  props: { padding: 16, spacing: 12 },
  children: [
    { type: "text", props: { text: "GPIO Control" } },
    { type: "button", props: { label: "Set High" } }
  ]
}

HTML-Like, Not HTML-Compatible

The syntax should feel ordinary: lowercase tags, children, props, event handlers, fragments, conditionals, and array maps. The contract should still be explicit: these tags are semantic aliases for native EMWaver primitives, not a promise to support the browser DOM.

Familiar aliases

Use normal-looking tags where they fit

<main>      -> Column
<section>   -> Card or Column section
<header>    -> Row/Column header region
<h1>-<h3>   -> Text with title roles
<p>         -> Text body
<button>    -> Button
<input>     -> TextField, Toggle, or Slider by type
<select>    -> Picker
Native extensions

Keep hardware widgets explicit

<Plot />
<SignalViewer />
<RegisterTable />
<PinGrid />
<SerialMonitor />

Hardware-specific elements should not be disguised as HTML. They are the reason EMWaver exists, so they deserve clear native names.

Runtime Pipeline

The in-between representation is not a competing UI system. It is the protocol that prevents every platform from needing its own parser, JS evaluator, component model, event mapper, and test snapshot format.

1

.emw JSX source

Readable component code, generated well by agents, editable by users.
2

JSX transform

Turns JSX syntax into calls like h(Column, props, children).
3

Script execution

Evaluates state, conditionals, maps, event closures, and hardware logic.
4

UI frame tree

Stable node types, props, children, handler tokens, metadata, and snapshots.
5

Native renderers

SwiftUI, Android, iOS, Windows, CLI snapshots, tests, and Agent context.

Primitive Set

Keep primitives at the shared native UI intersection. Let users write HTML-like aliases where possible, but normalize everything into a small native control vocabulary.

Category Primitives Notes
Layout main, section, div, row, grid, scroll, hr Aliases normalize to Column, Row, Grid, Scroll, Spacer, and Divider.
Display h1, h2, h3, p, span, card, tile, progress Good for hardware status, register values, diagnostics, and script state.
Input button, input type="checkbox", input type="range", input type="text", textarea, select DOM-style events such as onClick normalize to EMWaver handler tokens.
Hardware Plot, signal viewers, future oscilloscope/sampler widgets These are EMWaver-specific and should remain first-class, not forced into HTML metaphors.

Styling Contract

EMWaver should not become a styling framework. Support a small set of semantic props that are easy to render consistently on native platforms.

Allow

gap, padding, width, height, tone, variant, size, role, icon.

Avoid

CSS classes, arbitrary colors, custom fonts, box shadows, absolute positioning, z-index, and full style objects.

Default

Native platform appearance, consistent spacing, readable typography, and hardware-first information density.

<section tone="warning" padding={12}>
  <h3>Signal clipping</h3>
  <p>Lower gain or reduce sample period.</p>
  <button variant="primary" icon="waveform" onClick={restart}>
    Restart capture
  </button>
</section>

Migration Phases

This can be incremental. The existing render tree and native renderer do not need to be thrown away.

Phase 0

Freeze the current UI node contract

Document the node types, props, handler types, and snapshot shape as the private render protocol.

Foundation
Phase 1

Add JSX transform support

Bundle or precompile a small JSX transform so scripts can author components while emitting current nodes.

Low disruption
Phase 2

Create EMWaver component helpers

Expose Column, Button, Plot, and friends as JSX components backed by the existing UI builder.

User-facing
Phase 3

Add state and rerender rules

Define a small state API for script UI, including event-triggered rerender and timer-triggered rerender.

Runtime work
Phase 4

Migrate built-in scripts

Convert examples one by one. Keep compatibility with old builder-style scripts during the transition.

Validation
Phase 5

Update Agent generation

Teach the Agent to generate EMWaver JSX by default and use hardware primitive tools for exploration.

What Not To Do

Do not promise browser compatibility

HTML-like tags are aliases. Full HTML/CSS implies a browser layout engine, CSS cascade, DOM events, focus rules, and platform mismatch.

Do not make each native app parse source independently

That duplicates language semantics and creates cross-platform drift in errors, state, events, and snapshots.

Do not expose the tree as the user-facing API

The tree is the runtime protocol. JSX should be the thing users and agents learn.

Recommended naming: EMWaver JSX. It signals familiar syntax: close enough to normal JSX/HTML to read instantly, constrained enough to stay native and cross-platform.
