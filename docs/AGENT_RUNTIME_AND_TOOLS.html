EMWaver Agent

Agent script runtime and console model

The Agent is optional assistance for building, running, and debugging complete local JavaScript/JSX scripts.

Current Position

EMWaver stays local-first. The user can run scripts and control local hardware without an account or backend. The Agent is a paid, optional helper that can understand the current app context, write full .js scripts, run existing or ephemeral scripts locally, and inspect console output.

Goal 1

Write complete scripts

The Agent should create and edit runnable .js scripts that use normal imports, JSX-style UI, and EMWaver libraries like emw-jsx, emw-ui, emw-gpio, emw-spi, and emw-i2c.

Goal 2

Run scripts for evidence

The Agent should run JavaScript/JSX scripts that use public EMWaver libraries, then inspect console output and script status before it writes the final user-facing script.

Boundary

Optional network help

The Agent API is for intelligence and metered assistance. It is not a cloud control plane, script store, activation gate, or device ownership system.

The Shape

The source of truth for the user remains local files and the connected device. The model helps decide what to do, but the app owns execution, transport, safety boundaries, and script persistence.

Agent

Reasoning and authoring

Reads user intent, visible script context, runtime errors, and console output. Produces explanations, script patches, and ephemeral probe scripts.

→
Script runtime

Local script execution

The macOS app runs saved or ephemeral JavaScript/JSX against the selected local transport and captures console output.

→
Script artifact

Full local .js file

Once behavior is understood, the Agent writes or patches a complete script with imports, JSX UI, state, callbacks, and hardware library calls.

Why Scripts Instead of Primitive Tools

Rejected

Arbitrary eval

Letting the Agent evaluate fragments inside the live engine made the model guess the runtime scope. It would invent names, call stale APIs, or depend on variables that were never in scope.

Chosen

Real script runs

Saved and ephemeral scripts give the model an inspectable artifact. The native app owns local execution, session claims, console capture, and error reporting.

Decision: the macOS Agent should not operate hardware through UI button presses, arbitrary eval, or direct SPI/GPIO/ADC primitive tools. It should write or patch JavaScript/JSX, run it locally, then inspect console output and status.

Tool Surface

The macOS tool surface is script-focused. Higher-level behaviors such as "initialize CC1101" or "read an RFID card" should be captured as reusable local scripts or unsaved ephemeral probes first.

Tool group Tools Purpose
Workspace list_scripts, read_script, apply_patch_to_script Discover local scripts, inspect source, and apply user-approved edits.
Runtime run_script, run_ephemeral_script, stop_script, get_device_status, get_script_status, sleep Launch saved or unsaved scripts, stop them, inspect connection/script state, read console output, and wait before checking again.

Generated Script Style

The Agent should write normal-looking JavaScript modules. UI is JSX-like, hardware comes from focused EMWaver libraries, and scripts configure their own pins explicitly.

import { JSX, render } from "emw-jsx";
import { Column, Text, Button, LogViewer } from "emw-ui";
import { gpioPin, gpioMode, gpioWrite } from "emw-gpio";

const led = gpioPin({ port: "A", number: 0 });

function App() {
  return (
    <Column padding={16} spacing={12}>
      <Text font="title2">GPIO</Text>
      <Button onTap={() => {
        gpioMode(led, "output");
        gpioWrite(led, "high");
      }}>High</Button>
      <LogViewer />
    </Column>
  );
}

render(<App />);
No hidden bootstrap No hardcoded pin aliases No old UI globals No protocol scripts as examples

Agent Loop

  1. The user describes the target behavior: for example, blink a pin, read a sensor, initialize a radio, or build a control UI.
  2. The app supplies visible context: selected script, current error, connected board status, and recent console output.
  3. The Agent writes or patches a complete .js script using the public libraries.
  4. When it needs quick evidence, the Agent runs an unsaved ephemeral script against the selected local device.
  5. The app executes the script locally and captures console.log, console.warn, console.error, render state, and errors.
  6. The Agent calls get_script_status after any needed sleep and uses the console tail to continue.

Local App Versus Agent API

Local hardware
Runs through the native app and active local transport.
Agent API
Optional network call for model reasoning. Uses the public Continual MI/MGPT Agent API and API key when configured.
Failure mode
A connection error such as NSURLErrorDomain -1004 for a configured /api/mgpt/universes endpoint means the public Agent API host is not running or not reachable. It should not block local scripts or direct hardware control.
Script storage
Local scripts remain local files or app-local script records. The Agent can propose edits, but the user-facing artifact is still a local .js script.

Implementation Rules

Keep

Clear public surface

  • Use focused libraries such as emw-gpio, emw-spi, emw-i2c, emw-uart, and emw-ui.
  • Expose bundled read-only library scripts in the script list with a library/script differentiator.
  • Prefer explicit pins and bus configuration in user scripts.
  • Return script status and console output the Agent can reason over.
Avoid

Hidden or ambiguous behavior

  • No hidden script_bootstrap compatibility layer.
  • No automatic aliases like A0, IR_TX, or hardcoded SPI device pins in generic libraries.
  • No Agent hardware work through UI snapshots, button clicking, arbitrary JavaScript eval, or direct SPI/GPIO/ADC primitive tools on macOS.
  • No calls to MDL-only /backend-api/... routes from EMWaver clients.
  • No EMWaver account or cloud control dependency for local device access.

What This Enables

The Agent can become useful by producing real runnable scripts from the start. It can probe hardware with ephemeral JavaScript/JSX, inspect console output, and turn verified behavior into clean local files. That keeps EMWaver open-source and script-first while making the paid Agent valuable as a practical hardware assistant.
