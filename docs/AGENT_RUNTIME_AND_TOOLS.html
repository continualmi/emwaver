EMWaver Agent

Agent runtime and hardware tool model

The Agent is optional assistance for building complete local JavaScript scripts and for operating connected devices through typed hardware tools.

Current Position

EMWaver stays local-first. The user can run scripts and control local hardware without an account or backend. The Agent is a paid, optional helper that can understand the current app context, write full .js scripts, and directly exercise hardware through named tools such as SPI, I2C, GPIO, ADC, PWM, and UART operations.

Goal 1

Write complete scripts

The Agent should create and edit runnable .js scripts that use normal imports, JSX-style UI, and EMWaver libraries like emw-jsx, emw-ui, emw-gpio, emw-spi, and emw-i2c.

Goal 2

Interface with devices

The Agent should use hardware tools to perform transfers, probe registers, read pins, and validate behavior before it writes the final user-facing script.

Boundary

Optional network help

The Agent API is for intelligence and metered assistance. It is not a cloud control plane, script store, activation gate, or device ownership system.

The Shape

The source of truth for the user remains local files and the connected device. The model helps decide what to do, but the app owns execution, transport, safety boundaries, and script persistence.

Agent

Reasoning and authoring

Reads user intent, visible script context, runtime errors, and tool results. Produces explanations, script patches, and hardware tool calls.

→
Native tools

Typed hardware operations

The macOS app sends protocol packets through the active transport for SPI, I2C, GPIO, ADC, PWM, UART, sleep, and status checks.

→
Script artifact

Full local .js file

Once behavior is understood, the Agent writes a complete script with imports, JSX UI, state, callbacks, and hardware library calls.

Why Tools Instead of Running JS Snippets

Rejected

Arbitrary eval

Letting the Agent run ad hoc JavaScript made the model guess the live API surface. It would invent names, call stale APIs, or depend on script-local variables that were never in scope.

Chosen

Named hardware tools

Typed tools give the model a small exact interface. The native app owns EMW protocol packets, transport claims, timeouts, and error reporting.

Decision: the Agent should not operate hardware by pressing UI buttons or by evaluating guessed JavaScript. It should operate hardware through explicit primitive tools, then write or patch the final script for the user.

Tool Surface

The tools are intentionally low-level. Higher-level behaviors such as "initialize CC1101" or "read an RFID card" should be built by the Agent from primitives and then captured as reusable local scripts or libraries.

Tool group Tools Purpose
Workspace list_scripts, read_script, apply_patch_to_script Discover local scripts, inspect source, and apply user-approved edits.
Runtime run_script, stop_script, get_device_status, sleep Launch the full user script, stop it, inspect connection state, and wait for hardware settling.
GPIO gpio_mode, gpio_write, gpio_read Configure and inspect digital pins using explicit pin numbers or port/number mappings.
Analog and PWM analog_read, pwm_write Read ADC values and drive PWM outputs when supported by the active board firmware.
SPI spi_transfer Send raw SPI bytes with explicit chip-select handling and return received bytes.
I2C i2c_write, i2c_read, i2c_write_read Scan and communicate with I2C devices by address without hardcoded board aliases.
UART uart_write, uart_read Send and receive serial data through configured UART pins or board-supported UART channels.

Generated Script Style

The Agent should write normal-looking JavaScript modules. UI is JSX-like, hardware comes from focused EMWaver libraries, and scripts configure their own pins explicitly.

import { JSX, render } from "emw-jsx";
import { Column, Text, Button, LogViewer } from "emw-ui";
import { gpioPin, gpioMode, gpioWrite } from "emw-gpio";

const led = gpioPin({ port: "A", number: 0 });

function App() {
  return (
    <Column padding={16} spacing={12}>
      <Text font="title2">GPIO</Text>
      <Button onTap={() => {
        gpioMode(led, "output");
        gpioWrite(led, "high");
      }}>High</Button>
      <LogViewer />
    </Column>
  );
}

render(<App />);

Agent Loop

  1. The user describes the target behavior: for example, blink a pin, read a sensor, initialize a radio, or build a control UI.
  2. The app supplies visible context: selected script, current error, connected board status, and any user-approved hardware notes.
  3. The Agent calls hardware tools when it needs evidence from the device: read a pin, send SPI bytes, scan I2C, or verify a register.
  4. The app executes those tool calls locally against the active transport and returns structured results.
  5. The Agent writes or patches a complete .js script using the public libraries.
  6. The user runs the script locally. Script errors stay script errors; hardware tool failures stay Agent tool results.

Local App Versus Agent API

Local hardware
Runs through the native app and active local transport.
Agent API
Optional network call for model reasoning. Uses the public Continual MI/MGPT Agent API and API key when configured.
Failure mode
A connection error such as NSURLErrorDomain -1004 for a configured /api/mgpt/universes endpoint means the public Agent API host is not running or not reachable. It should not block local scripts or direct hardware control.
Script storage
Local scripts remain local files or app-local script records. The Agent can propose edits, but the user-facing artifact is still a local .js script.

Implementation Rules

Keep

Clear public surface

  • Use focused libraries such as emw-gpio, emw-spi, emw-i2c, emw-uart, and emw-ui.
  • Expose bundled read-only library scripts in the script list with a library/script differentiator.
  • Prefer explicit pins and bus configuration in user scripts.
  • Return structured tool results the Agent can reason over.
Avoid

Hidden or ambiguous behavior

  • No hidden script_bootstrap compatibility layer.
  • No automatic aliases like A0, IR_TX, or hardcoded SPI device pins in generic libraries.
  • No Agent hardware work through UI snapshots, button clicking, or arbitrary JavaScript eval.
  • No calls to MDL-only /backend-api/... routes from EMWaver clients.
  • No EMWaver account or cloud control dependency for local device access.

What This Enables

The Agent can become useful before it knows every final script API. It can probe hardware with stable tool schemas, then turn the verified behavior into clean local JavaScript. That keeps EMWaver open-source and script-first while making the paid Agent valuable as a practical hardware assistant.
