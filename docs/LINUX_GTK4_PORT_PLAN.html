EMWaver planning / Linux native app

Complete macOS-to-Linux GTK4 port plan

A native Linux desktop implementation that keeps EMWaver local-first, hardware-integrated, and self-contained while porting the macOS app's runtime, transport, script, firmware update, and optional Agent workflows to Rust + GTK4/libadwaita.

1. Decision and product boundary

Linux is back as a real native desktop application. The target framework is Rust + GTK4 + libadwaita, using Linux-native hardware APIs underneath and aligning with the desktop MCP direction once the runtime is ready.

UI

GTK4/libadwaita

GNOME-class native UI, adaptive layouts, settings, dialogs, toasts, device sheets, and polished desktop behavior.

Language

Rust

One language for UI glue, async runtime, protocol, USB, BLE, Wi-Fi, MCP bridge, helpers, tests, and packaging scripts where practical.

Hardware

Linux APIs

libusb/udev/serial/HID, BlueZ D-Bus, NetworkManager D-Bus when Wi-Fi configuration is needed, and native sockets/WebSockets.

Agent

Local-first + optional Agent

Scripts and firmware setup run through the native app, with optional Agent assistance when configured.

Do not restore the removed Linux architecture

This plan is a native app plan. The Linux app owns its own native runtime just like macOS owns its native runtime, and desktop agent access should come through the same in-app MCP model as the other desktop platforms.

2. Goals, non-goals, and parity target

Goals

  • Port the macOS app's local script execution model to Linux with equivalent UX and diagnostics.
  • Support USB, BLE, and Wi-Fi transports through native Linux APIs.
  • Support STM32 DFU and ESP32 serial flashing with bundled firmware images.
  • Implement the same optional Agent API-key workflow using the public /api/mgpt/... endpoint family.
  • Keep scripts and local app state on the user's machine.
  • Use reusable Rust crates so protocol/runtime logic can be tested without GTK.

Non-goals

  • Keep local scripts and firmware setup local-first.
  • No cloud script storage or sync by default.
  • Use local diagnostics and labels rather than product activation or ownership gates.
  • No dependency on ESP-IDF as an end-user runtime.
  • No system-wide daemon unless a narrowly scoped helper becomes unavoidable for firmware flashing permissions.
  • No Flatpak-only first release; hardware permissions are too important for the initial Linux target.

Core Rust crates

  • emwaver-protocol: shared packet encode/decode, opcodes, SysEx/superframe framing, simulator fixtures.
  • emwaver-session: script sessions, buffer isolation, device claims, heartbeats, stop/run lifecycle.
  • emwaver-transport: common transport traits and device identity/deduplication model.
  • emwaver-agent-client: public Agent API client, request context builder, streaming response support.

GTK app responsibilities

  • Render state, route user actions, display logs, show progress, and collect settings.
  • Own no hardware protocol constants directly.
  • Use async actions and message passing instead of blocking the GTK main loop.
  • Keep local scripts available even when Agent setup is missing or offline.

4. Runtime and state model

The Linux app should mirror the self-contained macOS model: one native app process owns discovery, selected device state, script execution, logs, Agent calls, and firmware update flows.

5. GTK4/libadwaita UI plan

Main window

AdwApplicationWindow with navigation split view: sidebar for scripts/devices/examples, main pane for Agent/editor/runtime, bottom or side log drawer.

Device sheet

Unified USB/BLE/Wi-Fi list, transport badges, UID suffix, firmware version, busy state, manual Wi-Fi host/port, provisioning and recovery actions.

Script editor

Use GtkSourceView for JavaScript/JSX syntax highlighting, run/stop controls, examples, local save/open, and inline active session status.

Agent panel

Conversation + action cards. Agent can propose scripts, modify local editor content, request hardware primitives, and explain errors.

Firmware update

Board-aware STM32 vs ESP32 update flow, logs, progress, port/device chooser, udev permission diagnostics, and recovery instructions.

Settings

Agent endpoint/key, discovery toggles, Wi-Fi defaults, log level, firmware image overrides, simulator mode, privacy/local data copy.

Native Linux feel target

Follow GNOME/libadwaita patterns: header bars, adaptive split views, toasts, preferences window, app menu actions, dark-mode support, keyboard shortcuts, and high-DPI icons. Avoid webviews for core UI.

7. Firmware/update flows

STM32

DFU update path

  • Detect run-mode board over the normal USB transport when possible.
  • Offer board-aware "Enter Update Mode" for STM32 only.
  • Use existing Rust DFU crates/helper from crates/ where possible.
  • Bundle canonical firmware/emwaver.bin.
  • Show udev permission errors with exact fix instructions.
  • Never require UID/account/activation to flash.
ESP32

Serial flashing path

  • Detect flash-capable serial ports and present candidate labels.
  • Bundle prebuilt bootloader, partition table, app, and OTA images from esp/build/.
  • Use a small Rust flasher or bundled esptool-compatible helper.
  • Present BOOT/RESET instructions where automatic bootloader entry is unreliable.
  • Do not bundle ESP-IDF or ask end users to run idf.py.
  • Return user to Run Mode instructions after flashing.

8. Desktop MCP integration

The Linux Agent/MGPT client is migration debt. Linux should eventually match the desktop direction: an in-app local MCP server backed by the native script runtime, console capture, filesystem scripts, and selected device transport.

AreaLinux behavior
EndpointLoopback Streamable HTTP, explicitly enabled by the user in the running native app.
CredentialGenerated local token/pairing secret; no EMWaver account or paid key for local hardware access.
ContextExpose selected script, errors, device type, transport kind, firmware version, capabilities, and recent logs through MCP resources/tools.
Hardware toolsMCP exposes named local primitives such as spi_transfer, gpio_read, gpio_write, and analog_read; the app runs them locally through the selected device.
Failure modeMCP server errors are local app errors and must never block script editor, device connection, or firmware update flows.

9. Linux permissions, security, and privacy

udev rules

Ship 99-emwaver.rules for supported VID/PID pairs and DFU/serial access. Installer should reload rules and explain group/logout requirements.

Polkit helper

Avoid by default. Consider only for narrow firmware operations that cannot be made reliable with udev rules. Keep source auditable and local-only.

Sandboxing

Initial packages should prioritize hardware access. Flatpak can be evaluated later with documented USB/BLE portal limitations.

  • Local scripts remain local files/app data.
  • Hardware UID is a local label/deduplication/diagnostic value only.
  • Wi-Fi control uses LAN/VPN reachability as the trust boundary.
  • No telemetry by default. If diagnostics export is added, it must be explicit and user-owned.

11. Validation plan

Automated validation

  • Protocol encode/decode unit tests.
  • Transport trait simulator tests.
  • Script session lifecycle tests: run, stop, timeout, busy device, reconnect.
  • Agent client request/response fixture tests with no real secrets.
  • Firmware flasher dry-run/argument construction tests.
  • GTK action/state tests where practical.

Manual hardware gates

  • STM32 USB run-mode script execution.
  • STM32 DFU flash and return-to-run-mode.
  • ESP32 BLE connection and script execution.
  • ESP32 Wi-Fi mDNS discovery and LAN script execution.
  • ESP32 manual IP/VPN script execution.
  • ESP32 serial flashing with prebuilt images.
  • Two-device bench: concurrent discovery, selected-device run isolation, busy-session behavior.

13. Key risks

Risk

Linux fragmentation

BlueZ, udev, ALSA, NetworkManager, and desktop packaging differ by distro. Start with Ubuntu LTS and Fedora as validation targets.

Risk

BLE reliability

BlueZ behavior can vary by adapter. Build strong diagnostics and keep USB/Wi-Fi first-class fallback paths.

Risk

Permissions

Hardware access fails if udev rules are missing. Installer and UI must detect and explain exact fixes.

Risk

Scope creep

Linux should not distract from V1 mobile-first priorities unless explicitly prioritized. Keep this as a staged native desktop port.
