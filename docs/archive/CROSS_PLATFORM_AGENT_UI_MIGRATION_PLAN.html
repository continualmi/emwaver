EMWaver migration plan

Make the Agent, UI, scripts, and libraries identical across iOS, Android, macOS, and Windows.

macOS is the behavior reference, but the product target is one EMWaver experience: the same Agent workflow, the same script authoring model, the same example/library/kernel separation, and the same visible .js/.jsx script surface on every supported platform.

Decision

Standardize EMWaver around normal JavaScript modules and JSX-style UI authoring while keeping native rendering and platform-native device transports. The user and the Agent should see one script API. The apps may use Swift, Kotlin, and C# internally, but they must implement the same runtime contract, the same generated UI node model, and the same script library exposure.

Agent

Identical user workflow

One chat/editor/run loop, one local context payload shape, one set of hardware primitive tools, and one explanation style across desktop and mobile.

UI

Identical script UI model

Scripts render JSX into an EMWaver UI tree. SwiftUI, Android, and Windows renderers consume that tree instead of each inventing UI semantics.

Scripts

Identical file surface

Bundled examples, kernel helpers, and libraries ship as readable .js and .jsx files, with clear imports and stable names.

Target Architecture

The migration is not a move to web views or React DOM. It is a move to a shared script/runtime contract that every native client implements. Native apps stay native; script code and Agent behavior become portable.

Authoring

.js and .jsx

Users and the Agent write normal modules with imports from EMWaver libraries.

Runtime

Shared JS contract

The same globals, imports, timers, events, errors, and lifecycle rules exist everywhere.

UI tree

Normalized nodes

JSX compiles to portable nodes for controls, layouts, plots, logs, forms, and status.

Native bridge

Platform renderers

SwiftUI, Android UI, and WinUI render the same node tree and event callbacks.

Agent

Context and tools

The Agent reads the same local context and calls the same named hardware primitives.

Script Surface

The old distinction between hidden bootstrap behavior and visible examples should be replaced with explicit layers. Users can inspect examples and libraries. Kernel internals are documented and stable, but not presented as tutorial code.

Authoring Standard

The Agent should generate the same style of script on every platform. Examples should teach that style directly. 

import { render, Column, Text, Button, Plot } from "emw-jsx";
import { gpioMode, gpioWrite } from "emw-gpio";
import { createSampler } from "emw-sampler";

const led = 13;
let on = false;
const sampler = createSampler({ channel: 0, hz: 500 });

async function toggle() {
  on = !on;
  await gpioMode(led, "output");
  await gpioWrite(led, on);
  render(<App />);
}

function App() {
  return (
    <Column gap={12}>
      <Text value={on ? "LED on" : "LED off"} />
      <Button title="Toggle" onPress={toggle} />
      <Plot title="A0" samples={sampler.samples} />
    </Column>
  );
}

render(<App />);

End User Parity

The platform-specific shell can differ where the operating system requires it, but the user's mental model must not change. A script created on macOS should open, run, render, fail, and be fixable by the Agent on iOS, Android, and Windows with the same names and behavior.

macOS

Use the current app as the reference for script list behavior, Agent context, run lifecycle, native UI rendering, transport selection, and local files.

iOS

Match the same script editor, examples, Agent workflow, runtime errors, JSX rendering, and device-session behavior within iOS storage and permission limits.

Android

Match the same script library, Agent context, USB/BLE/Wi-Fi device targeting, and JSX-native render bridge without Android-only script names or hidden APIs.

Windows

Match the same script session list, Agent API client, examples, file exposure, renderer semantics, and transport state through the WinUI host.

Agent Migration

The Agent is a cross-platform product surface, not a separate implementation per app. Each app should collect equivalent local context and expose equivalent actions.

Shared context payload

  • Current script text, file name, language mode, unsaved state, and selected example/library source.
  • Runtime error, console output, active UI tree summary, selected device, transport, firmware status, and recent hardware tool results.
  • Platform capabilities and permission state, expressed through shared capability names rather than platform-specific prose.

Shared Agent actions

  • Create or edit .js/.jsx scripts.
  • Explain runtime errors and hardware responses.
  • Call named hardware primitives such as gpio_read, gpio_write, analog_read, and spi_transfer.
  • Return runnable script patches instead of hidden app-specific commands.

Implementation Phases

Phase 1

Inventory and freeze the reference

Document macOS behavior for Agent, editor, run lifecycle, examples, library imports, file storage, session list, errors, and UI nodes. Turn that into a parity checklist.

Phase 2

Define the shared runtime contract

Finalize import resolution, module cache behavior, JSX transform rules, UI node schema, lifecycle callbacks, timers, cancellation, and error serialization.

Phase 3

Move examples and libraries

Promote visible examples and libraries to explicit .js/.jsx assets. Keep kernel helpers separated and remove tutorial-facing bootstrap leakage.

Phase 4

Port renderers and bridges

Bring iOS, Android, and Windows to the same renderer semantics and hardware primitive bridge as macOS, with fixture-based parity tests.

Phase 5

Unify Agent UX

Standardize the chat/editor/run affordances, local context shape, tool result display, script patch application, and API-key configuration behavior.

Phase 6

Validate cross-platform parity

Run the same scripts, UI fixtures, simulator sessions, and hardware smoke tests across macOS, iOS, Android, and Windows before treating the migration as complete.

Acceptance Criteria

1

A bundled JSX example opens and runs unchanged on macOS, iOS, Android, and Windows.

2

The same script imports emw-jsx, emw-ui, emw-gpio, emw-spi, and emw-i2c on every platform.

3

The same UI tree fixture renders equivalent controls, plots, labels, buttons, inputs, and event callbacks on every platform.

4

The same runtime error produces the same user-facing error shape and the same Agent context on every platform.

5

The Agent can create, edit, run, and fix a local script using the same visible workflow everywhere.

6

Examples, libraries, and kernel files are separated in app UI and repository layout.

7

Local hardware control remains account-free and does not require cloud activation, hosted relay, or script sync.

8

Parity tests cover runtime fixtures, UI node rendering, Agent context construction, simulator devices, and at least one real-board smoke path per platform.

Risks

Risk

Four script runtimes drift

Mitigation: keep shared fixtures in the repo and require each app to pass the same module, JSX, error, UI, and simulator cases.

Risk

Hidden bootstrap behavior survives

Mitigation: require every example to import the libraries it uses and keep kernel helpers out of tutorial-facing code.

Risk

Agent behavior becomes app-specific

Mitigation: version the Agent context schema and hardware primitive result schema, then test context construction per app.

Immediate Next Steps

  1. Create the runtime parity checklist from the current macOS Agent/UI/script behavior.
  2. Rename or reorganize bundled assets so examples, libraries, and kernel files are explicitly separated.
  3. Choose the canonical JSX transform path: runtime transform everywhere, precompile-on-save, or both with the same generated node output.
  4. Add shared fixtures for module imports, JSX output, UI node rendering, Agent context payloads, and simulator hardware responses.
  5. Port the iOS, Android, and Windows render/runtime bridges until the same examples pass unchanged.
