iOS Transport Session Lock Plan

Porting the macOS & Android transport session lock system to iOS.
One serial command bus, one lock, zero races — now for iOS.

1. The Problem

1.1 iOS: No Transport Session At All

The iOS USBManager never sends EMW_OP_TRANSPORT_SESSION (0x0B). ESP32 boards connected over BLE or Wi-Fi reject every operational command with BUSY (error 130). Discovery commands (VERSION, BOARD_GET, HARDWARE_UID) work, but script execution fails — SPI, GPIO, ADC, I2C, and sample/transmit are all blocked.

macOS and Android already have full transport session implementations: CONNECT → heartbeat loop (2s) → DISCONNECT. iOS has none of this.

⚠ iOS scripts on ESP32/BLE/Wi-Fi are broken by construction

Without a transport session, the firmware rejects operational commands. This applies to every BLE-connected ESP32-S3 board and every Wi-Fi-connected ESP32 board. STM32 over USB MIDI is unaffected (no session required).

1.2 iOS: No Command Lock

The iOS sendCommand() calls sendPacket(command) then awaitCommandResponse(timeout:) with no serialization. Two concurrent callers (script command + heartbeat) would race on the RX buffer. 

// iOS — current sendCommand (NO lock)
func sendCommand(_ command: Data, timeout: Int, deviceId: String) -> Data? {
    guard isConnected else { ... return nil }
    let session = withBufferQueueSync {
        let session = bufferSession(deviceId: deviceId)
        session.prepareCommandResponseWait()
        return session
    }
    sendPacket(command, deviceId: deviceId)      // fire-and-forget on midiQueue
    return session.awaitCommandResponse(timeout: timeout) { isConnected }  // poll loop
}

Even if we add a heartbeat to iOS without fixing this, the heartbeat and script command would steal each other's responses — the exact same predicate-race bug described in the macOS/Android plan.

6. User taps Stop → IOSScriptSession.stop() → manager.exitPreview()

Key insight: Step 4–5 has no transport session begin/end and no command lock. The fix hooks into steps 2 (begin session) and 6 (end session).

3. The Solution

3.1 Single serial command bus (iOS edition)

Same principle as macOS and Android: a DispatchSemaphore wraps the entire synchronous sendCommand body — prepare, send, wait. The heartbeat acquires the same lock. Only one command+response cycle is in flight at any time. 

// iOS — after fix
private let commandSemaphore = DispatchSemaphore(value: 1)

func sendCommand(_ command: Data, timeout: Int, deviceId: String) -> Data? {
    commandSemaphore.wait()
    defer { commandSemaphore.signal() }

    guard isConnected else { ... return nil }
    guard requireActiveDeviceSession(deviceId: deviceId, operation: "sendCommand") else { return nil }

    let session = withBufferQueueSync {
        let session = bufferSession(deviceId: deviceId)
        session.prepareCommandResponseWait()
        return session
    }

    sendPacket(command, deviceId: deviceId)

    return session.awaitCommandResponse(timeout: timeout) { isConnected }
}

Why a semaphore and not a serial queue

The existing sendCommand is already a synchronous call (it blocks waiting for a response). A serial queue would risk deadlocking if the heartbeat (also on midiQueue or its own queue) called sendCommand. A semaphore works across any calling queue — the heartbeat can be on any GCD queue and it just waits its turn.

3.2 Transport session lifecycle

Follow the macOS pattern exactly. The session is claimed when a script starts, held by a 2s heartbeat, and released when the script stops. 

1. Device connects (BLE/WiFi/USB)         → no session yet
2. User taps Run → IOSScriptSessionManager.run() → beginTransportSession()
3. Script runs → operational commands flow through lock
4. Heartbeat fires every 2s → keeps firmware session alive (5s timeout)
5. User taps Stop / script ends → endTransportSession()
6. DISCONNECT sent → session released
7. Heartbeat timer cancelled → no more heartbeat traffic

⚠ Heartbeat thread safety

The heartbeat timer fires on midiQueue. Since sendCommand is also called from midiQueue (via sendPacket), the semaphore must be used from the calling queue — not from midiQueue exclusively. The heartbeat's sendCommand call will block the midiQueue timer event until the lock is free. This is safe because:

  • No other work on midiQueue depends on the heartbeat completing first.
  • The lock is released in a defer block even on timeout.
  • The worst-case 2s script command timeout + heartbeat = 2s delay, well within the 5s firmware session timeout.

The source byte in the session command payload (byte 2 of [0x0B, sub, source]) must match what the transport layer stamps on cmd.source. The firmware rejects mismatches. iOS already tracks activeTransport as an enum with .usbMidi, .ble, and .wifi cases.

7. requiresTransportSession Logic

Follow the same logic as macOS and Android: 

private func requiresTransportSession(deviceId: String) -> Bool {
    // BLE and Wi-Fi always require a session.
    if activeTransport == .ble || activeTransport == .wifi {
        return true
    }
    // USB MIDI: ESP32 boards require a session; STM32 does not.
    // iOS doesn't currently track boardType as a property, but we can
    // inspect the last probed connected board.
    // Simpler: if activeTransport is .usbMidi and we have an ESP32 port name → true.
    if let portName = connectedPortName?.lowercased() {
        if portName.contains("esp32") || portName.contains("s3") || portName.contains("s2") {
            return true
        }
    }
    return false
}

iOS board type detection note

iOS currently doesn't store a dedicated connectedBoardType property like Android and macOS do. The simplest approach is to inspect the connected port name (as macOS does in inferBoardType(portName:)) or add a lightweight connectedBoardType: String? property populated during connect. For the initial implementation, the port-name heuristic is sufficient since all BLE/Wi-Fi connections already return true unconditionally.

10. Risk Assessment

RiskLikelihoodMitigation
Lock blocks UI thread Low sendCommand is already synchronous (blocks waiting for response). It's called from script execution (background), Wi-Fi provisioning (background), and heartbeat timer (midiQueue). No UI-thread callers.
Deadlock (semaphore) Low Single semaphore, no nested acquires. defer { signal() } guarantees release even on timeout. Heartbeat is the only other semaphore acquirer — no re-entrant paths.
Heartbeat delayed past 5s firmware timeout Very low Each command has max ~2s timeout. Heartbeat interval is 2s. Worst case: a max-duration script command holds the lock for 2s, heartbeat fires immediately after — 2s delay, well within 5s firmware timeout.
STM32 boards break None requiresTransportSession returns false for STM32. beginTransportSession returns true immediately without sending any session command. No behavior change for STM32.
sendCommandBestEffort race None sendCommandBestEffort is used only for identity checks during connection setup (before a script is running). It calls sendPacketBestEffortSync + manual awaitCommandResponse without the semaphore. This is intentional — it runs during connect/discovery when no script or heartbeat is active.
Wi-Fi provisioning during script run Low Wi-Fi provisioning calls sendCommand from a background queue. If a script is also running, both acquire the semaphore serially — no race. The Wi-Fi setup will simply wait for the current command to complete.
