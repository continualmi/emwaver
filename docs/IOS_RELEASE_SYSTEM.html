EMWaver — iOS Release System

The iOS release path is intentionally a single upload workflow. A signed App Store IPA uploaded to App Store Connect is processed by Apple, appears in TestFlight, and can later be selected for App Store review.

Workflow

WorkflowTriggersFastlane laneDestination
.github/workflows/ios-release-upload.yml
iOS Release Upload		 workflow_dispatch, plus tags matching ios-v* or emwaver-ios-v* fastlane ios release_upload App Store Connect processing; then TestFlight and App Store build selection
Why one workflow? App Store Connect does not need separate binaries for TestFlight and App Store review. Upload once, test the processed build in TestFlight, then promote/select that same build for App Store review manually in App Store Connect.
No unsigned IPA distribution. iOS builds must be signed and distributed through Apple channels. There is no public unsigned IPA equivalent to the Android APK download path.

Manual dispatch inputs

  • build_number — optional App Store build number override. Blank means auto-increment from the latest TestFlight build.
  • changelog — optional App Store Connect/TestFlight changelog text.

What the workflow does

  1. Checks out the repository and selects Xcode on the macOS runner.
  2. Installs the Fastlane bundle from ios/.
  3. Installs the App Store Connect API key, distribution certificate, and provisioning profile from the protected app-store GitHub Environment.
  4. Writes App Store export options for com.emwaver.EMWaver.
  5. Runs fastlane ios release_upload.
  6. Uploads the IPA as a GitHub Actions artifact for troubleshooting.

Fastlane lane

release_upload:

  1. Resolves the App Store Connect API key.
  2. Auto-increments the build number unless EMWAVER_IOS_BUILD_NUMBER is set.
  3. Configures App Store signing.
  4. Builds the Release scheme to build/ios/EMWaver.ipa.
  5. Validates that bundled default scripts are JavaScript assets and no legacy .emw files remain.
  6. Uploads the IPA to App Store Connect/TestFlight processing with skip_waiting_for_build_processing: true.

beta and app_store_upload remain compatibility aliases for release_upload.

Secrets required

The workflow uses the protected GitHub Environment named app-store with these secrets:

  • APP_STORE_CONNECT_API_KEY_ID
  • APP_STORE_CONNECT_API_ISSUER_ID
  • APP_STORE_CONNECT_API_KEY_BASE64
  • IOS_DISTRIBUTION_CERTIFICATE_BASE64
  • IOS_DISTRIBUTION_CERTIFICATE_PASSWORD
  • IOS_PROVISIONING_PROFILE_BASE64
  • IOS_PROVISIONING_PROFILE_NAME
  • IOS_KEYCHAIN_PASSWORD

Release procedure

  1. Run Actions → iOS Release Upload → Run workflow, or push an ios-v*/emwaver-ios-v* tag.
  2. Approve the protected app-store environment if required.
  3. Let Apple finish build processing in App Store Connect.
  4. Use the processed build in TestFlight for internal/external testing.
  5. When ready for production, open App Store Connect, select the same processed build for the version, confirm review/compliance details, and submit for review manually.

Local commands

scripts/ios-release.sh test
scripts/ios-release.sh archive
scripts/ios-release.sh release_upload
scripts/ios-release.sh release

ios/EMWaver/Info.plist sets ITSAppUsesNonExemptEncryption to false, so standard Apple export-compliance encryption prompts should be pre-answered for new builds.
