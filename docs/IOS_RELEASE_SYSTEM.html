EMWaver — iOS Release System

Reference for the two iOS distribution paths: TestFlight (internal/external testing) and App Store (production).

1. Overview

EMWaver iOS releases travel through two distinct paths, each served by its own GitHub Actions workflow. Unlike Android, there is no unsigned IPA distribution path — all iOS builds must be signed and go through Apple's official channels.

Path Workflow Trigger tags Artifact Signing Destination
TestFlight ios-testflight-release.yml ios-v* / emwaver-ios-v* Signed IPA Distribution cert TestFlight (internal only, no external groups)
App Store Upload ios-app-store-upload.yml Workflow dispatch only Signed IPA Distribution cert App Store Connect (draft, not submitted for review)
Key distinction: The TestFlight workflow supports both tag-based triggering and manual dispatch; it uploads to TestFlight only. The App Store workflow is dispatch-only (no tag trigger) and uploads a production candidate to App Store Connect as a draft — final submission for App Review must be confirmed manually in App Store Connect.
No unsigned IPA distribution. Unlike Android (which offers a direct APK download via GitHub Releases), iOS has no unsigned distribution path. All iOS builds are code-signed and must flow through Apple's App Store Connect infrastructure. External testers must be invited through TestFlight; there is no side-loading equivalent.

Fastlane lane: beta

  1. Resolves App Store Connect API key from environment
  2. Auto-increments build number (queries latest TestFlight build + 1, or uses explicit EMWAVER_IOS_BUILD_NUMBER)
  3. Configures manual code signing (team 9VA9KTUXXN, profile from IOS_PROVISIONING_PROFILE_NAME)
  4. Builds the Release scheme → signed IPA at build/ios/EMWaver.ipa
  5. Uploads to TestFlight with:
    • skip_waiting_for_build_processing: true — returns immediately, build appears in TestFlight when processing completes
    • distribute_external: false — only internal TestFlight group; external distribution must be enabled manually in App Store Connect
    • changelog — optional "What to Test" from EMWAVER_IOS_CHANGELOG

Characteristics

  • Signed — distribution certificate + App Store provisioning profile
  • Internal only — uploaded to TestFlight internal group; no external testers auto-invited
  • Auto build number — queries latest TestFlight build and increments unless overridden
  • Runner: macos-26
  • GitHub environment: app-store
  • Concurrency group: ios-testflight-release (no parallel runs)

How to release to TestFlight

1 Create a tag: git tag -a ios-v0.2.0 -m "TestFlight build 0.2.0"

2 Push the tag: git push origin ios-v0.2.0

3 The workflow runs automatically and uploads to TestFlight.

4 Or trigger manually from the Actions tab with workflow_dispatch and optional changelog.

5 Wait ~10–30 minutes for Apple to process the binary. It will appear under TestFlight → Builds.

6 Add internal testers in App Store Connect under TestFlight → Internal Testing.

7 For external testers, create an external group, add the build, and submit for Beta App Review.

3. Path 2 — App Store Upload (Production Candidate)

Workflow: ios-app-store-upload.yml

Triggers

  • workflow_dispatch only — no tag-based trigger. Manual dispatch with optional inputs:
    • build_number — override auto-incremented build number
    • release_notes — App Store release notes for this version (written to fastlane/metadata/en-US/release_notes.txt)
Why dispatch-only? Production App Store uploads are intentionally gated behind manual dispatch. There is no tag-triggered path to production. This prevents accidental production uploads from tag pushes and ensures every App Store candidate is shipped intentionally.

Fastlane lane: app_store_upload

  1. Resolves App Store Connect API key from environment
  2. Auto-increments build number (same logic as TestFlight lane: query + 1 or explicit override)
  3. Configures manual code signing
  4. Builds the Release scheme → signed IPA
  5. Calls internal upload_app_store_candidate which runs deliver (formerly ipa) with:
    • IPA: build/ios/EMWaver.ipa
    • Metadata path: fastlane/metadata
    • Screenshots path: fastlane/screenshots (skipped if the directory doesn't exist)
    • submit_for_review: false — the upload is a draft; you must submit for App Review manually in App Store Connect
    • force: true — overwrites existing metadata/screenshots
    • run_precheck_before_submit: false — skips precheck (run separately if needed)

Characteristics

  • Signed — same distribution certificate and App Store provisioning profile as TestFlight
  • Dispatch-only — no tag trigger; must be invoked manually from the Actions tab
  • Draft upload — the build appears in App Store Connect but is not submitted for review
  • Metadata included — uploads app metadata (description, keywords, promotional text, etc.) and screenshots alongside the binary
  • Runner: macos-26
  • GitHub environment: app-store
  • Concurrency group: ios-app-store-upload (no parallel runs)

How to upload a production candidate

1 Go to Actions → iOS App Store Upload → Run workflow

2 Optionally set a build_number and release_notes

3 Click Run workflow

4 Wait for the build and upload to complete (~15–40 minutes)

5 Open App Store Connect → [Your App] → App Store → iOS App

6 Select the uploaded build under the version entry

7 Complete any remaining store listing fields (privacy policy, export compliance, content rights)

8 Click Submit for Review

App Review is manual. App Store Connect does not automatically submit uploaded builds for App Review. After upload, you must go to App Store Connect, select the build for your version, and explicitly submit it. Apple's review process typically takes 24–48 hours.

4. Secrets required

Both workflows share the same app-store GitHub environment and the same set of secrets:

Secret Purpose How to obtain / generate
APP_STORE_CONNECT_API_KEY_ID App Store Connect API key ID (e.g. ABC123XYZ8) App Store Connect → Users and Access → Integrations → API Keys
APP_STORE_CONNECT_API_ISSUER_ID App Store Connect API issuer ID (UUID) App Store Connect → Users and Access → Integrations → API Keys
APP_STORE_CONNECT_API_KEY_BASE64 Base64-encoded .p8 private key file Download .p8 from App Store Connect, then: base64 -w0 AuthKey_ABC123XYZ8.p8
IOS_DISTRIBUTION_CERTIFICATE_BASE64 Base64-encoded distribution certificate (PKCS#12 / .p12) Export from Keychain Access as .p12, then: base64 -w0 distribution.p12
IOS_DISTRIBUTION_CERTIFICATE_PASSWORD Password for the .p12 archive Set when exporting the certificate from Keychain Access
IOS_PROVISIONING_PROFILE_BASE64 Base64-encoded App Store provisioning profile (.mobileprovision) Download from Apple Developer → Profiles, then: base64 -w0 EMWaver_AppStore.mobileprovision
IOS_PROVISIONING_PROFILE_NAME Provisioning profile display name (e.g. EMWaver AppStore) Apple Developer → Profiles → profile name
IOS_KEYCHAIN_PASSWORD Password for the temporary build keychain Choose any strong password; used only within the CI run, not a persistent secret

GitHub environment

Both workflows run in the app-store GitHub environment. This must exist in the repo at:

Settings → Environments → New environment → name: app-store

The eight secrets above should be scoped to this environment.

API key vs. Apple ID: The workflows use App Store Connect API keys (JSON Web Token-based authentication), not an Apple ID username/password. This avoids 2FA friction and session expiry. The API key must have Developer or App Manager role in App Store Connect.

6. TestFlight strategy

The TestFlight workflow uploads builds to the internal group only. Distribution to external testers is a manual step in App Store Connect.

Group Auto-distributed? Max testers App Review
Internal No — must add build to group manually 100 (App Store Connect team members) Not required
External No — must add build and submit for Beta App Review Up to 10,000 Required (first build only, then lighter reviews)

Promotion flow (TestFlight → App Store)

1 Push tag ios-v1.0.0 → build uploaded to TestFlight (internal)

2 Wait for processing (~10–30 min); install and test internally via TestFlight app

3 When ready for broader testing, create an External Testers group in App Store Connect

4 Add the build and submit for Beta App Review (first external build only)

5 After testing is complete, trigger ios-app-store-upload from Actions tab

6 In App Store Connect, select the build for the version and submit for App Review

Same build can go to production. If the TestFlight build is the exact binary you want in production, you can promote it directly in App Store Connect without running a separate App Store upload workflow. The separate app_store_upload workflow is useful when the production binary differs (e.g., different build number, updated metadata, or a fresh build from a different commit).

9. Signing architecture

iOS code signing in CI uses manual signing with a pre-generated distribution certificate and provisioning profile:

Signing assets

AssetFormatLifetimeNotes
Distribution certificate.p12 (PKCS#12, base64-encoded)1 year (renewable)Created in Apple Developer → Certificates → Distribution → iOS Distribution
Provisioning profile.mobileprovision (base64-encoded)1 year (renewable)Tied to the App ID, distribution certificate, and team
App Store Connect API key.p8 private key (base64-encoded)Until revokedUsed for upload, not for code signing

CI signing flow

  1. Create a temporary keychain with a one-time password
  2. Import the distribution certificate (.p12) into the keychain
  3. Set key partition list to allow codesign access without interactive prompt
  4. Install the provisioning profile to ~/Library/MobileDevice/Provisioning Profiles/
  5. The exportOptions.plist tells xcodebuild to use manual signing with the named profile for com.emwaver.EMWaver
Certificate expiration: Distribution certificates expire after 1 year. When a certificate expires, a new one must be generated, base64-encoded, and updated in the IOS_DISTRIBUTION_CERTIFICATE_BASE64 secret. The provisioning profile must also be regenerated against the new certificate and updated in IOS_PROVISIONING_PROFILE_BASE64. These secrets should be rotated together.

10. Comparison with Android release system

The iOS release system works alongside the Android system. Key structural differences:

iOSAndroid
Two separate workflows for TestFlight and App Store One workflow (android-play-release.yml) with a track parameter
No unsigned distribution path android-apk-release.yml → unsigned APK on GitHub Releases
Tag push → TestFlight only Tag push → internal track only; production via dispatch or promotion
App Store production: dispatch-only (no tag trigger) Play Store production: dispatch with track: production or promote in console
Ghost (macos-26) Runner (ubuntu-latest)
GitHub environment: app-store GitHub environment: play-store
8 secrets (ASC API key + signing cert/provisioning profile) 5 secrets (upload keystore + Google service account)
Tag prefixes: ios-v*, emwaver-ios-v* Tag prefixes: android-v*, emwaver-android-v* (play); emwaver-v* (apk)
Workflow files: All workflows live in .github/workflows/:
  • ios-testflight-release.yml
  • ios-app-store-upload.yml
  • android-apk-release.yml
