EMWaver Script Runtime Plan

Replace the bootstrap with visible, importable runtime libraries.

The current bootstrap is the whole script world: packet transport, protocol constants, hardware interfaces, UI nodes, JSX helpers, sampler APIs, and filesystem helpers. The target is to remove that monolith and ship those parts as normal read-only library files in the app.

Decision

Do not preserve script_bootstrap.emw as the long-term runtime owner. Replace it with ordinary bundled JavaScript files. Name the core file emw-kernel.emw, expose it as read-only runtime infrastructure in the main app UI, and move user-facing APIs into importable libraries such as emw-jsx.emw, emw-ui.emw, emw-gpio.emw, emw-spi.emw, emw-i2c.emw, and emw-components.emw. The emw- prefix is worth keeping in code because bare names like ui and hardware are too ambiguous once imports become normal.

Keep Internal

Kernel responsibility

Host primitive capture, packet send, timeout/timer basics, module loading, and safe runtime registration. This is the minimum surface every script needs.

Expose As Libraries

User-facing APIs

UI authoring, JSX, GPIO, SPI, I2C, UART, sampler helpers, and higher-level components should live in focused importable files that users and external tools can inspect.

Source Format

The new public script format should be JavaScript/JSX source carried in .emw files, not a mixed user-facing .emw/.js world. The .emw extension now marks these files as EMWaver programs: local hardware scripts, native UI, and platform imports that still remain readable JavaScript.

All EMW

No mixed bundle

Bundled examples, libraries, and kernel/runtime source should all move to .emw.

Type by metadata

Not by extension

The app should distinguish Example, Library, Kernel, User Script, and Signal by metadata and UI badges.

Extensionless imports

Stable import names

Scripts import "emw-gpio", "emw-spi", and "emw-jsx", not file paths with extensions.

script_bootstrap.emw is about 1,600 lines and currently mixes kernel, protocol, hardware, UI, JSX, sampler, and filesystem concerns.

const outputPin = pin({ port: "A", number: 0 });

function App() {
  return (

      <Text font="title2">GPIO</Text>

      </Button>
    </Column>
  );
}

render(<App />);

Hardware API Redesign

Opening up the runtime libraries should also fix the hardware API. Avoid global pin constants, imported mode constants, and board-specific aliases baked into the runtime. Scripts that need pins should declare them explicitly as GPIO numbers or port/number descriptors.

STM32 style

Port and pin

pin({ port: "A", number: 0 }) for boards that expose port groups.

ESP32 style

GPIO number

pin({ gpio: 4 }) for boards where GPIO number is the native identity.

No aliases

Script-owned mapping

No pin("IR_TX"), SPI pin aliases, or board-name shortcuts in the libraries. A script must map names to explicit pins itself.

import { pin, gpio } from "emw-gpio";
import { spi } from "emw-spi";

const irTxPin = pin({ port: "A", number: 0 });
const statusLedPin = pin({ gpio: 4 });
const displayBus = spi.open({
  sck: pin({ port: "B", number: 3 }),
  miso: pin({ port: "B", number: 4 }),
  mosi: pin({ port: "B", number: 5 })
});

gpio.mode(irTxPin, "output");
gpio.write(irTxPin, 1);

Mode and level values should be plain script values: "input", "output", "input_pullup", 0, and 1. They should not require imports.

Hardware libraries should be split by capability. pin and gpio come from emw-gpio; SPI comes from emw-spi; I2C/Wire comes from emw-i2c; UART comes from emw-uart; ADC comes from emw-adc; PWM comes from emw-pwm once those APIs are more than one-off GPIO-adjacent helpers.

Those libraries should not ship mutable board aliases. SPI, I2C, UART, PWM, ADC, and component helpers must take explicit pins supplied by the calling .emw script.

Import Model

Implement the module system in two layers: a small synchronous require(name) loader first, then a transform for ES-style import syntax once the loader is stable.

Step 1

CommonJS core

Runtime gets require(name), module.exports, exports, module cache, cycle guard, and module source lookup from moduleSources.

Step 2

Named imports

Transpile import { render } from "emw-jsx" into a require destructure before JSX transpilation/evaluation.

Step 3

Library registry

Resolve module specifiers such as emw-jsx, emw-ui, emw-gpio, emw-spi, and relative script paths such as ./lib/pins.emw.

Script List UX

Libraries should be visible and inspectable in the main app UI as read-only bundled files, alongside the other default scripts. They should not be hidden away, and the kernel should be named as kernel rather than disguised as a normal example. The list also needs a visible type differentiator so users can tell runnable examples from imported libraries before opening them.

Section Contents Behavior Visual Type
Examples Runnable bundled demos such as GPIO, Sampler, CC1101, JSX Hello. Open, run, make copy. Example badge, run button enabled.
Libraries emw-jsx.emw, emw-ui.emw, emw-gpio.emw, emw-spi.emw, emw-i2c.emw, emw-uart.emw, emw-adc.emw, emw-pwm.emw, emw-sampler.emw. Visible in the main script UI, read-only, inspect, make copy, import from scripts. Library badge, read-only lock, import name shown.
Kernel emw-kernel.emw and possibly emw-protocol.emw. Visible in the main script UI as runtime infrastructure, read-only, importable, not a hidden asset. Kernel badge, read-only lock, lower visual priority than examples.
User Scripts User-created scripts and libraries. Editable, runnable if script, importable if module/library. User badge plus script/library state.
Signals .raw and .txt signal files. Open in signal/text editor, not imported as script modules. Signal badge, no run/import affordance.

A bundled library should look like source code the user can inspect, but not like an example they are expected to run directly. The script row should expose its import specifier, for example import { pin, gpio } from "emw-gpio".

Default Script Cleanup

Splitting runtime APIs into visible libraries lets the bundled script list get smaller and more honest. The default examples should be scripts a user would actually run. Protocol surfaces, bus helpers, GPIO helpers, and sampler APIs should move into read-only libraries instead of appearing as runnable examples.

Current / Possible Script Target Reason
script_bootstrap.emw Delete after emw-kernel, emw-protocol, and libraries are active. Hidden monolith replaced by visible read-only runtime files.
gpio.emw Replace with emw-gpio.emw; keep only a small blink.emw or GPIO example if useful. pin and gpio are library APIs, not a standalone protocol script.
spi.emw, i2c.emw, uart.emw Replace with emw-spi.emw, emw-i2c.emw, emw-uart.emw. Bus APIs should be imported by real device examples, not shown as generic runnable scripts.
adc.emw, pwm.emw Replace with emw-adc.emw and emw-pwm.emw, or fold into a specific demo. Raw capability tests belong in libraries unless they demonstrate a useful workflow.
sampler.emw Move reusable capture/retransmit APIs to emw-sampler.emw; keep only a real sampler demo if it is product-worthy. Sampler helpers should be imported by scripts, while demos should stay focused on user workflows.
blink.emw, cc1101.emw, radio/RFID demos, JSX demos Keep as Examples, renamed to .emw with normal imports. These are runnable scripts that prove a real workflow or component.

The rule is: if a file mostly exposes protocol, transport, pin, bus, or reusable component behavior, make it a read-only emw-* library. If it shows a real end-user action like blinking a pin, configuring a CC1101, reading RFID, or rendering a JSX UI, keep it as an Example.

Migration Phases

Phase 0

Define visible library files

Replace the hidden bootstrap concept with bundled read-only files: emw-kernel.emw, emw-protocol.emw, emw-ui.emw, emw-jsx.emw, emw-gpio.emw, emw-spi.emw, emw-i2c.emw, emw-uart.emw, emw-adc.emw, emw-pwm.emw, and component libraries.

New model
Phase 1

Implement require

Wire moduleSources into the Apple runtime and support synchronous module evaluation/cache.

Runtime
Phase 2

Extract emw-jsx and emw-ui

Move JSX helper and UI node factories into visible libraries imported by examples.

UI
Phase 3

Extract hardware interfaces

Extract focused hardware libraries: emw-gpio for pins/GPIO, emw-spi for SPI, emw-i2c for I2C/Wire, emw-uart for Serial/UART, emw-adc for analog input, and emw-pwm for timer output. Remove built-in pin aliases and require explicit pins from each calling script.

Hardware
Phase 4

Extract sampler/components

Move Sampler APIs and reusable UI components into importable libraries.

Components
Phase 5

Expose library sections

Update the script list to show Examples, Libraries, Kernel, User Scripts, and Signals with clear badges, read-only state, import names, and run/import affordances.

UX
Phase 6

Prune default examples

Delete protocol/helper scripts that are replaced by emw-* libraries, including standalone GPIO, SPI, I2C, UART, ADC, PWM, sampler, and bootstrap-style files. Keep only product-worthy runnable examples such as Blink, CC1101, RFID/radio demos, and JSX examples.

Cleanup

Preferred naming: emw-kernel for the runtime core, emw-protocol for EMW command semantics, emw-jsx for JSX authoring, emw-ui for raw node factories, emw-gpio for pins/GPIO, emw-spi for SPI, emw-i2c for I2C/Wire, emw-uart for Serial/UART, emw-adc for analog input, and emw-pwm for PWM/timer output.
