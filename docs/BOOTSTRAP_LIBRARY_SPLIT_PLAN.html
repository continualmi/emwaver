EMWaver Script Runtime Plan

Split the bootstrap into a visible kernel and importable libraries.

The current bootstrap is the whole script world: packet transport, protocol constants, hardware interfaces, UI nodes, JSX helpers, sampler APIs, and filesystem helpers. The next step is to make those parts explicit, inspectable, and importable.

Decision

Do not hide the core runtime behind an anonymous bootstrap. Name it kernel.emw, expose it as read-only runtime infrastructure, and move user-facing APIs into ordinary importable libraries such as jsx.emw, ui.emw, hardware.emw, and components.emw.

Keep Internal

Kernel responsibility

Host primitive capture, packet send, timeout/timer basics, module loading, and safe runtime registration. This is the minimum surface every script needs.

Expose As Libraries

User-facing APIs

UI authoring, JSX, GPIO/SPI/I2C/UART interfaces, sampler helpers, and higher-level components should live in importable files that users and the Agent can inspect.

Current Bootstrap

script_bootstrap.emw is about 1,600 lines and currently mixes kernel, protocol, hardware, UI, JSX, sampler, and filesystem concerns.

Target Shape

kernel.emw

Host bridge, packet send, module loader, runtime registration, basic timers.

imports
protocol.emw

Opcode constants, packet builders, status/payload helpers.

hardware.emw

GPIO, pins, ADC, PWM, SPI, Wire/I2C, Serial/UART, device metadata.

ui.emw

UI node factories, event handler tokenization, render tree output, plot buffer registration.

jsx.emw

JSX helper, primitive tag bindings, render(<App />), later state helpers.

components.emw

Higher-level functional components: pin picker, board status, register cards, logs.

sampler.emw

Sampler APIs, signal file helpers, capture/retransmit helpers, sampler UI components.

Import Model

Implement the module system in two layers: a small synchronous require(name) loader first, then a transform for ES-style import syntax once the loader is stable.

Step 1

CommonJS core

Runtime gets require(name), module.exports, exports, module cache, cycle guard, and module source lookup from moduleSources.

Step 2

Named imports

Transpile import { render } from "jsx" into a require destructure before JSX transpilation/evaluation.

Step 3

Library registry

Resolve aliases such as jsx, ui, and relative script paths such as ./lib/pins.emw.

Script List UX

Libraries should be visible and inspectable. They should not be hidden away, and the kernel should be named as kernel rather than disguised as a normal example.

Section Contents Behavior
Examples Runnable bundled demos such as GPIO, Sampler, CC1101, JSX Hello. Open, run, make copy.
Libraries jsx.emw, ui.emw, hardware.emw, sampler.emw. Open read-only, inspect, make copy, import from scripts.
Kernel kernel.emw and possibly protocol.emw. Visible as runtime infrastructure, read-only, not runnable by default.
User Scripts User-created scripts and libraries. Editable, runnable if script, importable if module/library.
Signals .raw and .txt signal files. Open in signal/text editor, not imported as script modules.
