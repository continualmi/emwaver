Transport Session Protocol

How EMWaver clients claim exclusive command access to an ESP32 device.
Reference implementation: macOS MacUSBManager.swift & ESP32 firmware transport_session.c.

1. Why This Exists

ESP32 devices support multiple simultaneous transports: USB MIDI, BLE, and Wi-Fi. Without a session model, two apps (or two transports on the same device) could interleave commands, corrupt response routing, and break sampler streams.

The rule is simple: before any operational command (SPI, GPIO, ADC, sample, transmit, PWM, etc.) is accepted, the client must claim the transport session by sending a CONNECT command.

⚠ Android is currently missing this

The macOS app implements the full session lifecycle. The Android app does not send CONNECT, which is why cc1101.emw and other hardware scripts fail with error 130 (EMW_RESP_STATUS_BUSY = 0x82) on ESP32 boards. See §7 for the gap analysis.

3. Command Format

Every session command is a 3-byte payload inside the standard 18-byte command lane: 

Byte 0:   0x0B          (EMW_OP_TRANSPORT_SESSION)
Byte 1:   subcommand    (0x00 = STATUS, 0x01 = CONNECT, 0x02 = DISCONNECT, 0x03 = HEARTBEAT)
Byte 2:   source        (0x01 = USB, 0x02 = BLE, 0x03 = Wi-Fi)
Bytes 3-17: 0x00        (padding)

The response is a single status byte in the response lane: 

Byte 0:   0x80  →  OK
          0x82  →  BUSY (no session, or wrong source)

4.2 What Happens Without a Session

If a client sends an operational command (e.g. EMW_OP_SPI_XFER = 0x50) without first claiming the session: 

Client ➔ SPI XFER [50, 0A, 02, 02, F1, 00]   ← cc1101.emw VERSION probe
Firmware ➔ BUSY [82]                            ← transport_session_allows_command() returns false

JS kernel:  __emwAssertOk() sees status=0x82  →  throws "Device error: 130"
cc1101.emw:  catch  →  "Load failed: Device error: 130"

5. Firmware Decision Logic

The gate lives in esp/main/libraries/init.c, command_task(). Every command (USB, BLE, or Wi-Fi) hits this check: 

// init.c — command_task()
active_command_source = cmd.source;

if (!transport_session_allows_command(&cmd)) {
    send_binary_busy();               // → 0x82
    active_command_source = EMW_COMMAND_SOURCE_UNKNOWN;
    continue;                          // drop the command
}

transport_session_allows_command() in transport_session.c decides in this order: 

1. Expire stale claims (if last activity > 5s ago, set session to NONE)

2. Is this a CONNECT command?
   → Allowed if: no active session OR same source already owns it

3. Is this a STATUS command or a discovery command?
   (VERSION, HARDWARE_UID_GET, BOARD_GET, NAME_GET, WIFI_CFG_STATUS, TRANSPORT_SESSION_STATUS)
   → Always allowed — no session needed

4. Is there an active session?
   If NO  →  REJECT (BUSY)      ← THIS IS THE ERROR PATH

5. Does the command's source match the active session?
   If NO  →  REJECT (BUSY)

6. Update activity timestamp → ALLOW

5.1 Discovery Commands (Always Allowed)

These opcodes do not require a session. They are safe to query even when no session is active or another source owns it:

OpcodeValueCommand
EMW_OP_VERSION0x01Firmware version query
EMW_OP_HARDWARE_UID_GET0x08Read hardware UID
EMW_OP_BOARD_GET0x09Board type query
EMW_OP_NAME_GET0x04Device name query
EMW_OP_WIFI_CONFIG0x0AWi-Fi status only (sub=0x04)
EMW_OP_TRANSPORT_SESSION0x0BSession status query only (sub=0x00)

5.2 Session Timeout

The firmware auto-expires a session after 5000ms of inactivity (no command with a matching source). The macOS app sends a heartbeat every 2000ms to prevent this.

6. macOS App — Reference Implementation

6.1 Which transports need a session?

// MacUSBManager.swift
private func requiresTransportSessionInternal(deviceID: String) -> Bool {
    if deviceID.hasPrefix("ble:") || deviceID.hasPrefix("wifi:") {
        return true              // BLE and Wi-Fi always require session
    }
    return boardTypeForDeviceIDInternal(deviceID)?
        .lowercased()
        .hasPrefix("esp32") == true   // USB-to-ESP32 also requires session
                                       // USB-to-STM32 does NOT (no transport_session.c)
}

6.2 Script run entry point

// MacScriptSessionManager.swift — run()
guard device.beginScriptTransportSession(deviceID: targetID) else {
    return runFailure("Cannot run script: transport claim failed")
}

let manager = ScriptPreviewManager()
manager.attach(device: deviceBridge)
manager.render(script: source, ...)

// ... when script stops:
device.endScriptTransportSession(deviceID: targetID)

6.3 Session establishment

// MacUSBManager.swift — beginScriptTransportSession()
// 1. Determine source byte from transport type
let source: UInt8 = deviceID.hasPrefix("midi:")  ? 0x01   // USB
                  : deviceID.hasPrefix("ble:")   ? 0x02   // BLE
                  : deviceID.hasPrefix("wifi:")  ? 0x03   // Wi-Fi

// 2. Send CONNECT and wait for OK
let ok = sendTransportSessionCommandInternal(
    subcommand: EmwOpcode.transportSessionConnect,  // 0x01
    source: source,
    deviceID: targetID,
    timeoutMs: 1500,
    reportErrors: true
)

// 3. Start periodic heartbeat (every 2s)
startTransportSessionHeartbeatInternal(deviceID: targetID, source: source)

6.4 Heartbeat

// MacUSBManager.swift — startTransportSessionHeartbeatInternal()
let timer = DispatchSource.makeTimerSource(queue: midiQueue)
timer.schedule(
    deadline: .now() + 2.0,
    repeating: 2.0          // 2000ms < 5000ms firmware timeout
)
timer.setEventHandler {
    sendTransportSessionCommandInternal(
        subcommand: EmwOpcode.transportSessionHeartbeat,  // 0x03
        source: source,
        deviceID: deviceID,
        timeoutMs: 1000
    )
}

6.5 Teardown

// MacUSBManager.swift — endScriptTransportSession()
sendTransportSessionCommandInternal(
    subcommand: EmwOpcode.transportSessionDisconnect,  // 0x02
    source: claim.source,
    deviceID: targetID,
    timeoutMs: 1000
)
// Heartbeat timer cancelled
claim.heartbeatTimer.cancel()

7. Android Gap Analysis

What's missing

The Android app (USBService.java, AndroidBleProtocol.java, AndroidWiFiTransport.java) has zero references to EMW_OP_TRANSPORT_SESSION (0x0B), transport session connect/disconnect, or heartbeat logic.

When a script runs on an ESP32 board via BLE or Wi-Fi, every operational command (SPI, GPIO, ADC, etc.) is rejected with BUSY (130).

7.1 What needs to change

For BLE (the user's current transport), add these steps to AndroidBleProtocol.java:

// In onConnected() callback, after setActiveConnection():
// 1. Send CONNECT
byte[] connectCmd = {0x0B, 0x01, 0x02};  // TRANSPORT_SESSION, CONNECT, BLE source
byte[] response = host.sendCommand(connectCmd, 1500);

if (response == null || response.length == 0 || response[0] != (byte)0x80) {
    // session claim failed — show error, disconnect
}

// 2. Start heartbeat timer (every 2s)
// 3. On disconnect: send [0x0B, 0x02, 0x02]

The same pattern applies to AndroidWiFiTransport.java with source 0x03.

8. The 3-Byte Command in Context

You asked: "So the current system is we send EMW_OP_TRANSPORT_SESSION + EMW_TRANSPORT_SESSION_CONNECT + EMW_COMMAND_SOURCE_BLE — these three commands to establish a connection?"

Close — but it's one command with 3 bytes, not three separate commands. The packet is: 

[0x0B, 0x01, 0x02]
ByteNameValueRole
0Opcode0x0BTells the firmware "this is a transport session command"
1Subcommand0x01Tells the session module "I want to CONNECT"
2Source0x02Tells the session module "I am BLE"

The firmware already knows the source from the transport layer (cmd.source field is set by the BLE/USB/Wi-Fi receive path). Byte 2 must match that source for the CONNECT to succeed: 

// init.c — handle_transport_session_opcode()
case EMW_TRANSPORT_SESSION_CONNECT:
    if (requested_source != cmd.source     // byte 2 must match transport source
        || !transport_session_connect(cmd.source)) {
        send_binary_busy();
        return;
    }
    send_binary_ok(NULL, 0);

After that single 3-byte command succeeds, the session is active. The client then needs periodic HEARTBEAT commands [0x0B, 0x03, 0x02] to prevent the 5-second inactivity timeout from reclaiming the session.
