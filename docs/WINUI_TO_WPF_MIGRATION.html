EMWaver Windows
WinUI 3 → WPF Migration Plan

macOS Is the UX Reference

The macOS app is the canonical, validated, and actively maintained EMWaver desktop experience. It is the UX target for the WPF rebuild.

Every WPF design decision starts with: "What does macOS do?"

Do not port the WinUI 3 UI. The WinUI shell was built against a framework that models cannot work with effectively and an architecture (multi-page Frame navigation) that the macOS app has already proven unnecessary. The WinUI code is a liability, not a reference.

Decision: WinUI 3 was a mistake. WPF is the right native framework for this product.

Good news: ~70% of the codebase is pure C# with zero WinUI dependencies. Only the UI shell (XAML pages, dialogs, navigation) needs rebuilding.

Target: Windows 11, .NET 10, WPF, x64.

1. Why WinUI 3 Failed Us

ProblemImpact
Models are terrible at WinUI 3 Every UI change requires fighting both the framework and the agent. WinUI 3 has ~3 years of training data vs. WPF's ~20 years.
Can't build or test on this machine WinUI requires Windows 11 + Visual Studio. All Windows edits are blind — committed without compilation or verification.
SDK churn Windows App SDK previews break between versions. The project already pins Microsoft.WindowsAppSDK 1.8.260101001 and net10.0-windows10.0.22621.0.
Sparse community Few open-source examples, few Stack Overflow answers, few blog posts. When something doesn't work, you're on your own.
WinRT interop pain WinUI 3 wraps WinRT, adding layers of indirection for things WPF does directly via P/Invoke or standard .NET APIs.

2. Why WPF Is the Right Answer

StrengthWhy It Matters
20 years of training dataModels know WPF deeply. XAML + MVVM + data binding patterns are thoroughly represented in training corpora.
Same .NET ecosystemAll existing C# services compile as-is. Jint, SQLite, HttpClient, Windows.Devices.* — none of it changes.
XAML-basedMental model transfers from WinUI. Same markup language, different namespaces. Bindings, templates, styles all work the same way.
Mature toolingVisual Studio designer, live visual tree, hot reload — all battle-tested.
Direct Win32 interopP/Invoke, window handles, hardware access — no WinRT sandbox impedance for the things EMWaver actually does.
Predictable behaviorNo SDK churn. WPF in .NET 10 is the same WPF that shipped in .NET 5, 6, 7, 8.

4. Adaptation Detail: WindowsDeviceManager.cs

This is the only file in the entire services layer that touches WinUI. The change is surgical:

// BEFORE (WinUI)
using Microsoft.UI.Dispatching;
// ...
private DispatcherQueue? _ui;
internal void AttachUiDispatcher(DispatcherQueue dispatcherQueue) { _ui = dispatcherQueue; }
private void RunOnUi(Action action)
{
    var ui = _ui;
    if (ui == null) { action(); return; }
    _ = ui.TryEnqueue(() => action());
}

// AFTER (WPF)
using System.Windows.Threading;
// ...
private Dispatcher? _ui;
internal void AttachUiDispatcher(Dispatcher dispatcher) { _ui = dispatcher; }
private void RunOnUi(Action action)
{
    var ui = _ui;
    if (ui == null) { action(); return; }
    if (ui.CheckAccess()) { action(); }
    else { ui.Invoke(action); }
}

All other code in this 500+ line file — device enumeration, USB/BLE/Wi-Fi connect/disconnect, command/response protocol, DFU entry, Wi-Fi provisioning — is pure .NET + standard WinRT. None of it changes.

5.2 MVVM Pattern

WPF is MVVM-native. The existing code already uses INotifyPropertyChanged extensively in WindowsDeviceManager and FirmwareUpdateManager. WPF bindings consume this directly.

5.3 Navigation

WPF uses the same Frame + Page model as WinUI. The existing navigation pattern (ContentFrame.Navigate(typeof(ScriptsPage))) maps directly to WPF.

// WPF Frame navigation — identical API to WinUI
ContentFrame.Navigate(new Uri("Pages/ScriptsPage.xaml", UriKind.Relative));
// or
ContentFrame.Navigate(new ScriptsPage());

// Back navigation
if (ContentFrame.CanGoBack) ContentFrame.GoBack();

5.4 Theming

WPF doesn't have WinUI's built-in ElementTheme system. We use ResourceDictionary swapping:

// App.xaml.cs
public static void ApplyTheme(AppThemeMode theme)
{
    var dict = theme switch
    {
        AppThemeMode.Light => new ResourceDictionary { Source = new Uri("Themes/Light.xaml", UriKind.Relative) },
        AppThemeMode.Dark  => new ResourceDictionary { Source = new Uri("Themes/Dark.xaml", UriKind.Relative) },
        _ => null  // System: detect from Windows settings
    };
    if (dict != null)
        Application.Current.Resources.MergedDictionaries.Add(dict);
}

10. Risk Assessment

RiskSeverityMitigation
ScriptRenderer / ScriptPlotControl may have hidden WinUI deps Medium These render JSX trees into UI elements. The tree model is framework-agnostic but the rendering uses framework controls. WPF equivalents exist (ItemsControl, DataTemplate). Adaptation needed but pattern is the same.
ESP helper process launch may need adjustment Low System.Diagnostics.Process is the same in both. Paths to bundled binaries don't change.
SQLite path resolution None %LOCALAPPDATA%/EMWaver/agent-chat.sqlite is resolved the same way.
P/Invoke signatures (window icon) None WPF's Window.Icon property replaces the manual P/Invoke approach. Simpler, not harder.
BLE/Wi-Fi WinRT APIs None Windows.Devices.Bluetooth and System.Net.WebSockets are standard WinRT/.NET APIs. They work identically in WPF.
Can't validate on this machine Medium Still requires a Windows 11 workstation. But WPF edits will be more correct on first try due to model familiarity and framework stability. Less iteration needed.
