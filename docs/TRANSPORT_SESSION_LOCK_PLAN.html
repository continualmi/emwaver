Transport Session Lock Plan

Fixing macOS predicate fragility and implementing Android transport sessions.
One serial command bus, one lock, zero races.

1. The Problem

1.1 macOS: Predicate Routing Is Flimsy

The heartbeat runs on a background DispatchQueue.global(qos: .utility) thread while script SPI/GPIO commands run on the MIDI queue. Both call sendCommandInternal() concurrently. To keep responses from being stolen, macOS uses a predicate: 

// Heartbeat send:
sendCommandInternal(
    heartbeatData,
    responsePredicate: { lane in
        lane.first == 0x80 || lane.first == 0x81 || lane.first == 0x82
    }
)

// SPI send:
sendCommandInternal(
    spiData,
    responsePredicate: nil       // accepts anything
)

This fails catastrophically when:

  1. Script sends SPI_XFER. SPI waiter's predicate is nil → accepts any response lane.
  2. Heartbeat fires on background thread, sends [0x0B, 0x03, source].
  3. Both commands go out to the device (order depends on thread scheduling).
  4. Device responds to heartbeat with [0x80] — first byte is 0x80.
  5. SPI waiter's predicate (nil) matches. SPI waiter consumes the heartbeat OK.
  6. Device responds to SPI with [0x80, rxByte1, rxByte2].
  7. Heartbeat waiter's predicate matches. Heartbeat waiter consumes the SPI data.
  8. Script gets garbage or wrong-length response. Breaks.

⚠ Predicate is defense-by-coincidence

It only works if responses arrive in exactly the same order as sent, AND responses are distinguishable. SPI responses start with 0x80 — exactly what the heartbeat predicate accepts. It's broken by construction.

1.2 Android: No Session At All

The Android app never sends EMW_OP_TRANSPORT_SESSION. BLE/WiFi-connected ESP32 devices reject all operational commands with error 130 (BUSY). This is the bug the user is hitting with cc1101.emw.

1.3 Android: No Response Routing Either

DeviceBufferSession.awaitCommandResponse() is raw polling — it grabs the next >= 0x80 status packet with no filtering: 

// Android — no predicate, just grabs next 0x80+ packet
public byte[] awaitCommandResponse(int timeoutMs) {
    while (...) {
        byte[] pkt = nextRxPacketData();
        if (pkt != null && (pkt[0] & 0xFF) >= 0x80) {
            return Arrays.copyOf(pkt, PACKET_SIZE_BYTES);
        }
    }
}

Even if we add transport session to Android without fixing this, two concurrent sendCommand callers (script + heartbeat) would steal each other's responses.

2. The Solution: Single Serial Command Bus

Principle: Only one command+response cycle can be in flight at any time. A single lock serializes all sendCommand calls — script and heartbeat alike. The heartbeat simply acquires the same lock and waits its turn.

3. Lock Placement

The lock wraps the entire synchronous sendCommand body — prepare, send, wait: 

// Pseudocode — both platforms
sendCommand(command, timeout):
    lock.acquire()
    try:
        prepareResponseWaiter()
        writePacket(command)
        response = awaitResponse(timeout)
        return response
    finally:
        lock.release()

// Heartbeat — same lock
sendHeartbeat():
    lock.acquire()
    try:
        prepareResponseWaiter()
        writePacket([0x0B, 0x03, source])
        response = awaitResponse(1000)
        if response[0] != 0x80:
            handleSessionExpiry()
    finally:
        lock.release()

3.1 macOS lock location

Add a DispatchSemaphore(value: 1) or serial DispatchQueue inside MacUSBManager.sendCommandInternal(). The heartbeat already routes through sendTransportSessionCommandInternal → sendCommandInternal, so it automatically acquires the same lock.

3.2 Android lock location

Add a private final Object commandLock = new Object() to USBService. Wrap sendCommand() and the heartbeat send in synchronized (commandLock).

4.3 Why macOS doesn't need more changes

The heartbeat already calls sendTransportSessionCommandInternal → sendCommandInternal. Once sendCommandInternal is locked, the heartbeat automatically joins the serial bus. The predicate can stay or be removed — the lock makes it redundant.

5.2 Heartbeat retry / session expiry

If a heartbeat response is not 0x80 (or times out), the session is considered expired. The timer is cancelled, the session is cleared, and an error is surfaced. The next operational command will get BUSY — same behavior as if no session existed.

5.3 Why the lock doesn't break the heartbeat

The heartbeat fires every 2s. The firmware session timeout is 5s. A worst-case script command takes <2s (timeout). Even if the heartbeat is delayed behind a max-duration command for the full 2s, it still fires well within the 5s window. No session expiry unless the device is actually unreachable.

The source byte in the session command payload (byte 2 of [0x0B, sub, source]) must match what the transport layer stamps on cmd.source. The firmware rejects mismatches.

8. Session Scope

Session = script execution, not device connection

A session is established when a script starts running and released when it stops. The device can be connected without a session — discovery commands still work. Only operational commands (SPI, GPIO, ADC, sample, PWM, I2C, UART, transmit) require it.

8.1 Lifecycle

1. Device connects (BLE/WiFi/USB)         → no session yet
2. User hits Run                          → beginTransportSession()
3. Script runs                            → operational commands flow through lock
4. Heartbeat fires every 2s               → keeps firmware session alive (5s timeout)
5. User hits Stop / script ends           → endTransportSession()
6. DISCONNECT sent                        → session released
7. Heartbeat timer cancelled              → no more heartbeat traffic

10. Risk Assessment

RiskLikelihoodMitigation
Lock blocks UI thread Low All sendCommand calls already happen on background threads (MIDI handler, BLE callback, script executor). No UI thread access.
Deadlock Low Single lock, no nested acquires. Lock is released in finally block even on timeout.
Heartbeat delayed past 5s timeout Very low Each command has max ~2s timeout. Heartbeat interval is 2s. Worst case: 2s command + heartbeat = 2s delay, well within 5s.
macOS predicate removal breaks existing behavior None Lock makes predicates redundant. They can stay or be removed — no behavior change.
STM32 boards break None STM32 firmware doesn't compile transport_session.c. Session commands get no response or error. requiresTransportSession() returns false for STM32.
