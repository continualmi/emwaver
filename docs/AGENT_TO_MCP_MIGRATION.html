Architecture Migration · Continual MI

Agent → MCP, Filesystem Scripts (Desktop)

EMWaver stops shipping a bespoke cross-platform Agent on every platform. The running desktop app becomes an MCP-addressable hardware engine: it connects to devices, lists JavaScript scripts from disk, runs them, and renders UI — and any external agent (e.g. Claude) drives it through tools. The Agent and MGPT integration are removed everywhere. iOS and Android stay, minus the agent and minus on-disk scripts.

Status: planned Date: 2026-06-06 Supersedes: AGENT_API.md · MGPT_UNIVERSE_AGENT_PLAN.md · AGENT_RUNTIME_AND_TOOLS.html Scope: agent removal (all) · MCP + disk scripts (desktop)

Why this change

The bespoke Agent never fit. We tried it listening to and driving the UI, then gave it named hardware primitives (spi_transfer, gpio_*, analog_read), and the honest endpoint was always the same: the Agent should just run JavaScript on the engine the app already has. But building and maintaining that agent loop across five platforms is a money pit — and the agent loop is the single fastest-commoditizing layer in the whole stack. We don't have a moat there, and we shouldn't pay to compete in it.

The resolution: don't ship an agent. Ship the door. Embed an MCP server in the running app that exposes the engine's tools, and let any frontier agent reach in. The hard, irreplaceable work — firmware, the JS engine, the device transports, the script-defined UI — already exists and runs once in the app. We stop porting intelligence and start exposing a socket.

This is repackaging, not deletion of value. The hardware primitive tools and the script-run/console model don't die. They stop being "Agent tools" and become the MCP tool surface. What dies is the agent runtime, chat UI, and MGPT key plumbing — the expensive-to-replicate parts.

The four moves

01

Nuke the Agent

Remove every Agent chat surface and runtime client on all five platforms. No in-app assistant anywhere.

02

Nuke MGPT keys & API

Drop EMWAVER_AGENT_API_KEY, EMWAVER_AGENT_ENDPOINT, the universe model, and all /api/mgpt/responses client code — everywhere.

03

Scripts live on disk · desktop

On desktop, scripts are real files in a live folder the app lists and runs. Mobile still has user scripts — imported into the app's storage rather than watched from an external folder.

04

MCP door · desktop

The desktop app embeds an MCP server external agents drive. Mobile can't host a listening endpoint, so it ships without MCP.

All five platforms stay. macOS, Windows, Linux, iOS, and Android remain shipping apps. What changes everywhere is the Agent/MGPT removal. What changes only on desktop is the new on-disk script model and the embedded MCP server.

The MCP server carries no logic — no planning loop, no orchestration. It opens a local SSE/HTTP endpoint, lists tools, and routes each call straight into the engine that's already in the app. That is why it ports cheaply: the platform-specific intelligence stays in the engine; only a socket shim is per-platform.

Design discipline: write the MCP tool definitions for a model to consume — clear names, structured returns, and error strings that explain how to recover. The endpoint lives inside the app, next to the engine; do not split it into a separate backend process and reintroduce an IPC hop to the device.

What the EMWaver app is now

The product statement collapses to four verbs:

·

Connect

Discover and connect to devices over USB / BLE / Wi-Fi.

·

List

Show the JavaScript scripts found in real filesystem locations.

·

Run

Execute scripts on the engine and surface console.log output / status — on every platform.

·

Render

Draw the native UI panels scripts define with JSX-style syntax.

Everything the old Agent wanted to do — read, patch, run, inspect — is now done either by the human at the UI or by an external agent through the identical MCP tools. There is one execution path, not two.

Console output parity. console.log capture was recently added on macOS only; bring it back to all five platforms (Windows, Linux, iOS, Android included). It's load-bearing now: the engine's console is what the human sees in the UI and what the MCP run_script tool returns to an external agent. Same capture, same format, every platform.

Filesystem scripts · desktop only

On desktop, scripts stop being an app-internal/bundled-only concept. They are .js files in real, user-visible directories on the machine. The app watches those locations, lists what it finds, and runs them. This makes scripts editable by any editor, versionable with git, and trivially reachable by both the human and the MCP agent.

Mobile uses import, not a live folder. iOS and Android can still bring in user scripts — via the document picker, the Files app, the share sheet, Android's Storage Access Framework, or iCloud/Drive — but the file is copied into the app's own storage and managed there (export back out is fine too). What mobile can't do is sit on top of a real, user-visible folder as a live source of truth. So the desktop-specific part is the external-folder model (edit in any editor, git, hot-reload, same paths the MCP server reads), not the ability to have user scripts.
From:  assets/default-scripts/*.js   bundled into each app, app-internal listing
To:    a real on-disk scripts directory the app indexes, e.g.
         ~/EMWaver/scripts/            user scripts (listed + run)
         <app-shipped defaults>        seeded into the directory on first run
       → app lists from disk, hot-reloads on change, runs on the engine
       → MCP `list_scripts` / `run_script` operate on the same paths
Single source of truth. The human UI and the MCP list_scripts/run_script tools must resolve the same filesystem locations. No divergence between "what the app shows" and "what the agent can run."

Open detail to settle: the canonical default directory per OS (macOS / Windows / Linux), whether multiple roots are supported (a shipped-defaults root + a user root), and how the bundled emw-* libraries (emw-spi.js, emw-gpio.js, etc.) are resolved from on-disk scripts.

The MCP surface

The tools are the former Agent hardware primitives plus script lifecycle. Indicative set — names final-able later:

write_scriptCreate/patch a script file on diskfilesystem

gpio_read / gpio_writeGPIO line read/writedevice transport

Transport: local SSE/HTTP on a loopback port so an already-running app can be dialed into (no spawning, no flashing). Auth for the local endpoint is a follow-up detail (loopback-only is the v1 default).

Execution checklist

  1. Rip out the Agent — all five apps. Delete agent chat UI + runtime clients in macOS, Windows, Linux, iOS, and Android. Remove agent menu entries, settings panes, and key-entry flows everywhere.
  2. Rip out MGPT — all five apps. Delete EMWAVER_AGENT_API_KEY/EMWAVER_AGENT_ENDPOINT config, universe persistence, and all /api/mgpt/responses client code. Strip "Agent"/"conversation" copy.
  3. Move desktop scripts to disk. Define per-OS script root(s) for macOS/Windows/Linux, seed shipped defaults + emw-* libs on first run, index + hot-reload from disk, resolve library imports from on-disk scripts. Leave mobile script handling as-is.
  4. Console parity across all five. Bring console.log capture (macOS-only today) to Windows, Linux, iOS, and Android. Same capture point and format, surfaced in the UI and returned by the MCP run_script tool.
  5. Add the desktop MCP server. Embed an in-process MCP server (loopback SSE) in the desktop apps exposing the tool set above, routing into the existing engine and transports. Model-friendly schemas + error strings.
  6. Re-home the docs. Archive the agent/MGPT docs; rewrite CURRENT.md/PLANNING.md for the no-agent + desktop MCP + desktop filesystem-scripts model (all five apps retained); update the public website docs.
  7. Verify end-to-end. Drive a real board through Claude→MCP (run_script + spi_transfer) on macOS/Windows/Linux, confirming parity with the human UI path. Confirm iOS/Android still run scripts and connect to devices with the agent removed.

Product / business note

Killing the agent doesn't mean "no product." The platform (firmware, engine, instant electronics, JS scripting, MCP door) becomes the adoption engine — given away to get reach and to harvest traces of real EMWaver workflows that nobody else can collect. Monetization rides on top:

now

Open platform + MCP

Frontier agents drive it. Adoption + trace collection. No bespoke agent to maintain.

next

Fine-tuned MGPT model

Trained on collected traces. Pitch isn't "smarter than Opus" — it's cheaper, faster, controllable.

wedge

Bundled distribution

Served through Continual Plus / Max for the MDL base. Inference-as-commodity is thin; bundled + domain-tuned is the moat.

Tension to hold honestly: if a frontier model drives the CLI/MCP well, the fine-tuned model is a cost/integration play, not a capability play — plan the model product as cheaper & bundled, not as the only thing that works.
