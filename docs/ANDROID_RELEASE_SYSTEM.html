EMWaver — Android Release System

Reference for the three distribution paths: APK (direct), Play Store internal/testing, and Play Store production.

1. Overview

EMWaver Android releases travel through three distinct paths, each served by its own GitHub Actions workflow and triggered by different tag conventions:

Path Workflow Trigger tags Artifact Signing Destination
APK android-apk-release.yml emwaver-v* Signed APK Upload key GitHub Releases (prerelease)
Play Store · internal/testing android-play-release.yml android-v* / emwaver-android-v* Signed AAB Upload key Google Play (internal / alpha / beta tracks)
Play Store · production android-play-release.yml android-v* / emwaver-android-v* Signed AAB Upload key Google Play (production track)
Key distinction: The APK workflow and the Play Store workflow use different tag prefixes so they never collide. Pushing emwaver-v1.0.0 builds a signed direct-install APK for GitHub Releases. Pushing android-v1.0.0 or emwaver-android-v1.0.0 triggers the signed Play Store upload.

2. Path 1 — Direct APK Distribution

Workflow: android-apk-release.yml

Triggers

  • workflow_dispatch — manual trigger with a tag input (default: emwaver-preview)
  • Push of any tag matching emwaver-v*

What it does

Stage firmware
→
./gradlew assembleRelease
→
Attach APK to GitHub Release

Characteristics

  • Signed — built from app-release.apk and verified with apksigner
  • Protected secrets — requires the Android upload keystore secrets from the play-store GitHub Environment
  • Prerelease — GitHub Release is always marked prerelease: true
  • Output name: EMWaver-android.apk
  • Works with any pre-existing GitHub Release tag (does not create the release itself)

How to release an APK

1 Create a tag: git tag -a emwaver-v0.2.0 -m "Preview release 0.2.0"

2 Push the tag: git push origin emwaver-v0.2.0

3 The workflow runs automatically and attaches the APK to the release.

4 Or trigger manually from the Actions tab with workflow_dispatch and specify the tag.

Prerequisite: The GitHub Release must already exist for the tag. The workflow attaches to an existing release — it does not create one. Create the release first from the GitHub Releases UI or push the tag with a release creation step.

3. Path 2 — Play Store (internal / testing / production)

Workflow: android-play-release.yml

Triggers

  • workflow_dispatch with optional inputs:
    • version_code — overrides build.gradle versionCode
    • version_name — overrides build.gradle versionName
    • changelog — Google Play changelog for this release
    • track — Play Store track (default: internal)

Secrets required

Secret Purpose How to generate
ANDROID_KEYSTORE_BASE64 Upload signing key (JKS, base64-encoded) base64 -w0 emwaver-upload.jks
ANDROID_KEYSTORE_PASSWORD Keystore password Set during key generation
ANDROID_KEY_ALIAS Key alias (e.g. emwaver-upload) Set during key generation
ANDROID_KEY_PASSWORD Key password Set during key generation
GOOGLE_PLAY_SERVICE_ACCOUNT_JSON Google Play service account JSON key Google Cloud Console → Service Accounts → Create key (JSON)

GitHub environment

The workflow runs in the play-store GitHub environment. This must exist in the repo at:

Settings → Environments → New environment → name: play-store

The five secrets above should be scoped to this environment.

Fastlane lane: beta

The workflow calls bundle exec fastlane android beta in the android/ directory. The Fastlane configuration (android/fastlane/Fastfile) does:

  1. Validates that all four signing environment variables are present and the keystore file exists
  2. Validates that the Google Play service account JSON key exists
  3. Runs gradle bundleRelease to build the signed AAB
  4. Calls upload_to_play_store with:
    • Package: com.emwaver.emwaverandroidapp
    • Track: GOOGLE_PLAY_TRACK env var (default: internal)
    • Release status: draft
    • Metadata/changelogs: skipped (not uploaded)

4. Track strategy

The android-play-release.yml workflow accepts a track input that maps directly to Google Play tracks. The default is internal.

Track Workflow input Typical use
Internal testing internal (default) Fast internal distribution, up to 100 testers, no review
Closed testing (alpha) alpha Invited testers, pre-production validation
Open testing (beta) beta Public opt-in testing, larger audience
Production production Full public release on Google Play
Recommended flow: Tag push → internal by default → promote manually in Play Console through alpha → beta → production. Or use workflow_dispatch with explicit track: production when the build is ready. All uploads arrive as draft and must be manually submitted/reviewed in Play Console before they go live.

Promotion example (step by step)

1 Push tag android-v1.0.0 → AAB uploaded to internal track (draft)

2 Review and promote to alpha in Play Console

3 After alpha validation, promote to beta

4 When ready for production, either:

  • Promote from beta to production in Play Console, or
  • Run a new workflow_dispatch with track: production

6. Tag convention reference

PrefixWorkflow triggeredArtifactDestination
emwaver-v*android-apk-release.ymlSigned APKGitHub Releases
android-v*android-play-release.ymlSigned AABGoogle Play (track from env)
emwaver-android-v*android-play-release.ymlSigned AABGoogle Play (track from env)
Important: These prefixes must not overlap. The APK workflow fires on emwaver-v*. The Play Store workflow fires on android-v* and emwaver-android-v*. A single tag like emwaver-v1.0.0 will only trigger the APK workflow, not the Play Store one.

7. How to generate an upload signing key

If you haven't created the upload key yet:

keytool -genkey -v \
  -keystore emwaver-upload.jks \
  -keyalg RSA \
  -keysize 2048 \
  -validity 10000 \
  -alias emwaver-upload \
  -storetype JKS

Then encode it for the GitHub secret:

base64 -w0 emwaver-upload.jks

Register the public certificate of this key in the Play Console under Setup → App Integrity → App signing. Google will use this to verify uploads and then re-sign with the Play App Signing key for distribution.
